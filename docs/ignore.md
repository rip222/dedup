# Ignore-rule layers

dedup consults a four-layer stack to decide whether a file should be
read during a scan. The implementation and precedence rules live in
`crates/dedup-core/src/ignore.rs`.

## The four layers

### Layer 1 — Hard-coded physical gates

Always on, unless `--all` is passed.

- **`.git/` contents** — any path whose components contain a
  `.git` segment is skipped (including submodule `.git/`).
- **Size cap** — files strictly larger than `scan.max_file_size`
  bytes (default 1 MiB, configurable in [config.md](config.md)) are
  skipped.
- **Binary content sniff** — the first 512 bytes of each file are
  scanned for a NUL byte. If any is present the file is treated as
  binary and skipped.

Layer 1 cannot be re-enabled by a `.dedupignore` whitelist — these
are physical constraints, not policy.

### Layer 2 — `.gitignore`

Delegated to the [`ignore`](https://crates.io/crates/ignore) crate,
which respects:

- Per-directory `.gitignore` files.
- `.git/info/exclude`.
- The user's global git excludes (`core.excludesFile`).

Disabled by `--no-gitignore`, and also disabled implicitly by
`--all`.

### Layer 3 — Built-in defaults

A curated list of patterns hard-coded in
`crates/dedup-core/src/ignore.rs::BUILT_IN_PATTERNS`:

- **Minified / bundled / generated by filename:**
  `*.min.js`, `*.min.css`, `*.bundle.js`, `*.generated.*`
- **Lock files:**
  `Cargo.lock`, `package-lock.json`, `yarn.lock`, `poetry.lock`,
  `pnpm-lock.yaml`
- **Vendored directories:**
  `vendor/`, `third_party/`, `third-party/`

Plus a **header scan** — the first five lines of each text file are
scanned for the case-sensitive markers `@generated` or
`AUTO-GENERATED`. Matches are skipped.

Disabled by `--all`.

### Layer 4 — `.dedupignore`

A per-repo file at `<repo_root>/.dedupignore` using
`.gitignore` syntax, parsed via the same `ignore` crate.

Applied **unconditionally** — `--all` does *not* disable this
layer. This gives you a guaranteed way to keep a file out of the
scan regardless of other flags.

## Precedence

In order of evaluation:

1. **Layer 1** runs first. Anything it rejects is gone — no later
   layer can whitelist it (binary files can't be parsed, oversized
   files can't fit in memory, `.git/` isn't source).
2. **Layer 2** runs on paths layer 1 allowed, via the walker.
3. **Layer 4 beats Layer 3.** An explicit `!pattern` re-include in
   `.dedupignore` overrides a layer-3 built-in match. This is the
   documented escape hatch if you want to scan a lock file or a
   vendored directory.
4. **Layer 3** runs last (when `--all` is not set).

## CLI flag summary

| flag | disables |
|---|---|
| `--no-gitignore` | Layer 2 only. |
| `--all` | Layers 1, 2, 3. Layer 4 still applies. |

## `.dedupignore` syntax

Identical to `.gitignore`:

```
# Skip everything under scratch/
scratch/

# …but not keep.tmp.
!scratch/keep.tmp

# Scan the repo's lock file anyway.
!Cargo.lock
```

Parse failures are logged at `debug` level and the file is treated
as absent — they do not derail a scan.
